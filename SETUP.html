Cómo levantar Serubix en local

Esta guía cubre todo lo necesario para clonar el repositorio desde GitHub y tener el proyecto funcionando en tu máquina en menos de 15 minutos.

Next.js 15 Express 5 Node 20 PostgreSQL 16 Docker opcional

✓ Requisitos previos

Asegúrate de tener instaladas las siguientes herramientas antes de continuar.

Node.js
v20 LTS o superior
Necesario para el frontend y el backend. Recomendado instalar con nvm.
npm
v10 o superior
Viene incluido con Node.js. Gestiona las dependencias de ambos proyectos.
Docker Desktop
v24 o superior
Sólo necesario para la Opción A. Incluye Docker Compose v2.
PostgreSQL
v16
Sólo necesario para la Opción B. En la Opción A lo levanta Docker.
Git
v2.x
Para clonar el repositorio desde GitHub.
Cuenta OpenAI
API key con créditos
Necesaria para la herramienta Text to Speech. Sin ella el TTS devuelve 503.
Google OAuth
Client ID + Secret
Para el login con Google. Opcional si solo usas email/contraseña.

Verificar versiones instaladas

Terminal 
node --version    # debe ser v20.x o superior
npm --version     # debe ser 10.x o superior
docker --version  # opcional — para Opción A
git --version

⚠ Ficheros no incluidos en el repositorio

El repositorio no contiene ficheros con secretos ni configuración local. Deberás crearlos antes de arrancar el proyecto.

📄
/.env
Contiene todas las variables de entorno del proyecto: credenciales de base de datos, claves de API, secretos de Auth.js y configuración de despliegue. Está excluido del repositorio por seguridad.
Cómo crearlo: copia .env.example que sí está en el repo y rellena los valores.
📂
/Backend/storage/
Directorio donde el backend guarda los audios generados por Text to Speech. Se crea automáticamente la primera vez que se genera un audio, no hay que crearlo manualmente.
Con Docker, este directorio está montado como volumen tts_storage para que los archivos persistan entre reinicios.
📂
/FrontEnd/.next/
Directorio de compilación de Next.js. Se genera automáticamente al ejecutar npm run build o npm run dev.
📂
/Backend/dist/
Salida de la compilación TypeScript del backend. Se genera automáticamente al ejecutar npm run build. En desarrollo (npm run dev) no es necesario compilar.

⚙ Variables de entorno

El proyecto usa un único fichero .env en la raíz del repositorio. Cópialo desde la plantilla:

Terminal — raíz del proyecto 
cp .env.example .env

Luego edita .env y rellena cada valor:

Variable Obligatorio Descripción y ejemplo
DOMAIN opcional Dominio del VPS en producción. En local puede dejarse vacío o poner localhost. DOMAIN=serubix.com
POSTGRES_USER requerido Usuario de PostgreSQL. POSTGRES_USER=serubix
POSTGRES_PASSWORD requerido Contraseña de PostgreSQL. Pon cualquier valor en local. POSTGRES_PASSWORD=mi_password_segura
POSTGRES_DB requerido Nombre de la base de datos. POSTGRES_DB=serubix
DATABASE_URL requerido Cadena de conexión completa de PostgreSQL. Con Docker usa el hostname db; sin Docker usa localhost. DATABASE_URL=postgresql://serubix:password@db:5432/serubix
JWT_SECRET requerido Secreto para firmar los tokens JWT del backend. Mínimo 32 caracteres. openssl rand -base64 32
AUTH_SECRET requerido Secreto de Auth.js v5 (NextAuth). Mínimo 32 caracteres. openssl rand -base64 32
AUTH_GOOGLE_ID opcional Client ID de Google OAuth. Necesario solo si quieres login con Google. Obtén las credenciales en console.cloud.google.com → APIs y Servicios → Credenciales. URI de redirección autorizada: http://localhost:3000/api/auth/callback/google AUTH_GOOGLE_ID=xxxx.apps.googleusercontent.com
AUTH_GOOGLE_SECRET opcional Client Secret de Google OAuth. AUTH_GOOGLE_SECRET=GOCSPX-xxxx
OPENAI_API_KEY opcional Clave de API de OpenAI. Necesaria para la herramienta Text to Speech. Sin ella el endpoint devuelve 503. Obtén una en platform.openai.com/api-keys. OPENAI_API_KEY=sk-proj-xxxx
NEXT_PUBLIC_API_URL requerido URL de la API visible desde el navegador. Con Docker y nginx: http://localhost/api. Sin Docker: http://localhost:4000. NEXT_PUBLIC_API_URL=http://localhost:4000
FRONTEND_URL opcional URL del frontend usada por el backend para CORS. FRONTEND_URL=http://localhost:3000
GITHUB_REPOSITORY opcional Solo necesaria para publicar imágenes Docker en GHCR. GitHub Actions la rellena automáticamente en CI. GITHUB_REPOSITORY=tu-usuario/TFM
Importante — DATABASE_URL en local vs Docker
Si usas Docker (Opción A), el backend accede a la base de datos usando el hostname db (nombre del servicio en Docker Compose): postgresql://serubix:password@db:5432/serubix
Si levantas el backend sin Docker (Opción B), usa localhost: postgresql://serubix:password@localhost:5432/serubix

🐳 Opción A — Con Docker Recomendada

Esta es la opción más sencilla. Docker Compose levanta automáticamente la base de datos, el backend, el frontend y el nginx reverse proxy. No necesitas instalar PostgreSQL ni configurar nada adicional.

Prerequisito
Docker Desktop debe estar ejecutándose antes de continuar. Puedes verificarlo con docker info.
1
Crear el fichero .env
Si aún no lo has hecho, copia la plantilla y rellena los valores según la sección anterior.
Terminal — raíz del proyecto 
cp .env.example .env
# Edita .env con tu editor favorito y rellena los valores
⚠ Ajusta DATABASE_URL
Para Docker, asegúrate de que DATABASE_URL use el hostname db (no localhost):
DATABASE_URL=postgresql://serubix:password@db:5432/serubix
2
Construir y levantar los contenedores
La primera vez tarda unos minutos porque construye las imágenes. Las siguientes veces es inmediato.
Terminal — raíz del proyecto 
docker compose up -d --build
Esto levanta 4 contenedores: db (PostgreSQL), backend (API), frontend (Next.js) y nginx (reverse proxy).
3
Ejecutar las migraciones de base de datos
Las migraciones crean todas las tablas necesarias en PostgreSQL. Solo es necesario hacerlo la primera vez (o cuando haya nuevas migraciones).
Terminal 
docker compose exec backend npx prisma migrate deploy
4
Poblar la base de datos con los datos iniciales
El seed crea los planes Free (5.000 caracteres TTS/mes) y Pro (ilimitado). Sin esto, el registro de usuarios fallará.
Terminal 
docker compose exec backend npm run db:seed
5
Verificar que todo está en marcha
Comprueba el estado de los contenedores:
Terminal 
docker compose ps
Todos los contenedores deben aparecer con estado Up. Abre http://localhost en el navegador.

Comandos de gestión Docker

Gestión de contenedores 
docker compose up -d          # Levantar (sin reconstruir)
docker compose down           # Parar y eliminar contenedores
docker compose down -v        # Parar + borrar volúmenes (⚠ borra la BD y audios)
docker compose logs -f        # Ver logs de todos los servicios
docker compose logs -f backend # Ver logs solo del backend
docker compose restart backend # Reiniciar un servicio

⚡ Opción B — Sin Docker (desarrollo manual)

Útil si prefieres mayor control sobre cada proceso o no tienes Docker instalado. Necesitarás PostgreSQL corriendo localmente.

Paso 1 — Instalar dependencias

Terminal 
# Instalar dependencias del backend
cd Backend
npm install

# Instalar dependencias del frontend
cd ../FrontEnd
npm install

Paso 2 — Configurar PostgreSQL

Asegúrate de tener PostgreSQL 16 corriendo en tu máquina. Crea la base de datos y el usuario:

psql — como superusuario 
CREATE USER serubix WITH PASSWORD 'mi_password';
CREATE DATABASE serubix OWNER serubix;
GRANT ALL PRIVILEGES ON DATABASE serubix TO serubix;
Ajusta DATABASE_URL
En el fichero .env, usa localhost en lugar de db:
DATABASE_URL=postgresql://serubix:mi_password@localhost:5432/serubix
Y también: NEXT_PUBLIC_API_URL=http://localhost:4000

Paso 3 — Migraciones y seed

Terminal — carpeta Backend/ 
cd Backend
npm run db:migrate   # Crea las tablas en la BD
npm run db:seed      # Inserta los planes Free y Pro

Paso 4 — Levantar backend y frontend

Necesitarás dos terminales abiertas simultáneamente:

Terminal 1 — Backend (puerto 4000) 
cd Backend
npm run dev
Terminal 2 — Frontend (puerto 3000) 
cd FrontEnd
npm run dev

Con esta opción no hay nginx. Accede directamente a http://localhost:3000 (frontend) y http://localhost:4000 (API).

✕ Solución de problemas comunes

🔴 El contenedor backend falla al arrancar con error de BD
El backend arranca antes de que PostgreSQL esté listo. Docker Compose lo gestiona con un healthcheck, pero en ocasiones es necesario esperar o reiniciar el backend:
Terminal
docker compose restart backend
🔴 Error "relation does not exist" o tablas no encontradas
Las migraciones de Prisma no se han aplicado todavía. Ejecuta:
Terminal
docker compose exec backend npx prisma migrate deploy
docker compose exec backend npm run db:seed
🔴 El registro de usuario falla con error de plan
Los planes Free y Pro no existen en la base de datos. El seed no se ha ejecutado:
Terminal
docker compose exec backend npm run db:seed
🔴 El frontend no puede conectar con la API (errores de red)
Comprueba que NEXT_PUBLIC_API_URL en .env apunta a la URL correcta según la opción que estás usando:
  • Con Docker + nginx: NEXT_PUBLIC_API_URL=http://localhost/api
  • Sin Docker: NEXT_PUBLIC_API_URL=http://localhost:4000
Reconstruir tras cambiar .env
Las variables NEXT_PUBLIC_* se incrustan en el build de Next.js. Si cambias NEXT_PUBLIC_API_URL debes reconstruir el frontend: docker compose up -d --build frontend
🔴 Login con Google no funciona
Verifica que en Google Cloud Console, la URI de redirección autorizada incluye exactamente:
  • http://localhost:3000/api/auth/callback/google (sin Docker)
  • http://localhost/api/auth/callback/google (con Docker + nginx)
🔴 El TTS devuelve error 503
La variable OPENAI_API_KEY no está configurada en .env o no tiene créditos. Añade una clave válida y reinicia el backend:
Terminal
docker compose restart backend
🔴 Puerto 80 o 3000 ya en uso
Otro proceso está ocupando el puerto. Identifícalo y ciérralo:
macOS / Linux
lsof -i :80    # Qué proceso usa el puerto 80
lsof -i :3000  # Qué proceso usa el puerto 3000
lsof -i :4000  # Qué proceso usa el puerto 4000
🔴 Contenedores con estado "Exited" o "Restarting"
Revisa los logs del contenedor que falla para ver el error concreto:
Terminal
docker compose logs --tail=50 backend
docker compose logs --tail=50 frontend
docker compose logs --tail=50 db
¿Todo en marcha?
Si la landing carga en http://localhost, el registro de usuarios funciona y el endpoint http://localhost/api/health devuelve {"status":"ok"}, el proyecto está correctamente configurado.
